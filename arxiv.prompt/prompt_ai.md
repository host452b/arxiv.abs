# AI/LLM Prompt 核心技巧总结

> 基于 arXiv 2024 年 CS 类 1478 篇 AI/LLM Prompt 相关论文摘要的系统性分析整理。
> 按照统一的结构化格式整理，便于直接应用和对比选择。
> 整合后的内容涵盖 **40 项核心技巧**及其 9 套组合应用方案。

---

## 1. 链式思维提示（Chain-of-Thought Prompting）

| 项目 | 内容 |
|------|------|
| **模板** | 在提示中加入"Let's think step by step"或提供带推理过程的示例。例如：`Q: {问题}\nA: Let's think step by step. First, ... Then, ... Therefore, the answer is {答案}` |
| **原理** | 引导模型在输出答案前先生成中间推理步骤，将复杂问题分解为可管理的子步骤。Google DeepMind 发现即使不用显式提示，通过非贪心解码路径也能触发 CoT 推理，说明 CoT 能力是模型内在的。 |
| **来源** | Google DeepMind，2024（arXiv:2402.10200），112 篇相关论文 |
| **适合任务** | 数学推理、逻辑推理、多步计算、复杂问答、代码调试 |

---

## 2. CoT 分隔符增强（CoT Separator Enhancement）

| 项目 | 内容 |
|------|------|
| **模板** | 在 CoT 的每个示例末尾添加分隔符标记。例如：`示例1: {推理过程} → {答案}\n---\n示例2: {推理过程} → {答案}\n---\n问题: {新问题}` |
| **原理** | 分隔符帮助模型区分不同推理示例的边界，减轻密集 prompt 的信息过载。实验表明在 GSM8K、AQuA 等复杂推理任务上，分隔符显著提升 CoT 效果。 |
| **来源** | 2024（arXiv:2402.10645） |
| **适合任务** | 多示例 CoT 场景、复杂推理、数学应用题 |

---

## 3. 少样本 / 上下文学习（Few-Shot / In-Context Learning）

| 项目 | 内容 |
|------|------|
| **模板** | 在提示中提供 1-5 个输入-输出示例。例如：`示例1: 输入: {x1} → 输出: {y1}\n示例2: 输入: {x2} → 输出: {y2}\n现在请处理: 输入: {x_new} → 输出:` |
| **原理** | 通过示例向模型展示任务格式和期望输出模式，无需微调即可引导模型行为。研究发现即使一个 few-shot 示例也能显著降低 prompt 敏感性（POSIX 指标），且示例选择对效果影响极大。 |
| **来源** | 综合研究，245 篇相关论文，含 POSIX（arXiv:2410.02185） |
| **适合任务** | 文本分类、命名实体识别、格式化输出、翻译、任何需要格式对齐的任务 |

---

## 4. 自动演示生成（Auto-Demo Prompting）

| 项目 | 内容 |
|------|------|
| **模板** | 在批量处理中，将模型前面的输出作为后续问题的示例。例如：先用 0-shot 回答 Q1，将 Q1 的结果作为 Q2 的 few-shot 示例，依次累积。 |
| **原理** | 利用模型自身产生的高质量输出作为动态 demonstration，无需人工编写示例。在批量推理场景中减少退化问题，有时甚至超越单条 prompt 效果。 |
| **来源** | 2024（arXiv:2410.01724） |
| **适合任务** | 批量推理、零样本效果差的任务、冷启动场景 |

---

## 5. 混合专家提示（Mixture-of-Expert Prompts）

| 项目 | 内容 |
|------|------|
| **模板** | 将示例按语义聚类，为每个簇生成专门的 instruction，推理时路由到最匹配的专家 prompt。 |
| **原理** | 单一 prompt 难以覆盖任务的所有子类型。通过将示例聚类后为每个语义区域搜索专用指令，形成 MoE 式 prompt 组合，在多个基准上实现约 81% 的胜率。 |
| **来源** | 2024（arXiv:2407.00256） |
| **适合任务** | 多样化输入的分类任务、复杂 NLP 任务、需要差异化策略的场景 |

---

## 6. 提示链 vs 逐步提示（Prompt Chaining vs Stepwise Prompt）

| 项目 | 内容 |
|------|------|
| **模板** | **提示链**：分别发送 Draft → Critique → Refine 三次调用。**逐步提示**：单次提示要求"先起草，再自我评价，最后修改"。 |
| **原理** | 将复杂任务拆分为多次独立调用（提示链）优于在一个 prompt 中要求模型模拟多步流程（逐步提示）。研究表明逐步提示可能只是"模拟"而非真正执行迭代优化。 |
| **来源** | 2024（arXiv:2406.00507），162 篇 chaining/decomposition 相关论文 |
| **适合任务** | 文本摘要、内容创作、代码审查、任何需要迭代优化的生成任务 |

---

## 7. 元提示（Meta-Prompting）

| 项目 | 内容 |
|------|------|
| **模板** | 一个 LLM 充当"指挥官"，分解任务后调用多个独立"专家"LLM 实例处理子任务，最后整合验证结果。例如：`你是一个任务协调者。请将以下任务分解为子任务，分别由不同专家处理，然后整合结果。任务: {复杂任务}` |
| **原理** | 利用一个模型作为 conductor 进行任务分解、子任务分发和结果整合。在 GPT-4 上平均比标准 prompting 高 17.1%，比动态专家 prompting 高 17.3%。 |
| **来源** | Stanford, 2024（arXiv:2401.12954） |
| **适合任务** | 复杂多步任务、跨领域综合分析、需要多种能力协同的场景 |

---

## 8. 提示格式化（Prompt Formatting）

| 项目 | 内容 |
|------|------|
| **模板** | 在保持内容相同的前提下，使用不同格式（纯文本、Markdown、JSON、YAML）包装提示。例如用 JSON 格式化输入：`{"task": "{任务描述}", "input": "{输入内容}", "constraints": ["{约束1}", "{约束2}"]}` |
| **原理** | 格式本身对 LLM 性能影响极大。GPT-3.5-turbo 在代码翻译任务上仅因格式变化就可波动约 40%。更大模型（GPT-4）更鲁棒，但仍不能假定一种固定模板永远最优。 |
| **来源** | 2024（arXiv:2411.10541），165 次引用 |
| **适合任务** | 所有任务 — 特别是代码生成、结构化输出、分类任务 |

---

## 9. 自动提示优化（Automatic Prompt Optimization）

| 项目 | 内容 |
|------|------|
| **模板** | 使用自动化框架（如 PromptWizard、SAMMO）对 instruction 和示例进行迭代优化。流程：生成候选 prompt → 评估 → 反思 → 改进 → 循环。 |
| **原理** | 人工编写 prompt 效率低且不稳定。PromptWizard 通过自演化循环在 45 个任务上达到顶级性能，且大幅减少 API 调用。SAMMO 将 prompt 表示为符号程序，搜索编译时变换。但研究也发现 LLM 作为优化器存在误归因错误的风险。 |
| **来源** | Microsoft PromptWizard（arXiv:2405.18369）、SAMMO（arXiv:2404.02319）、arXiv:2402.02101，106 篇相关论文 |
| **适合任务** | 需要大规模部署的 prompt、生产环境调优、RAG pipeline 优化 |

---

## 10. 提示压缩（Prompt Compression）

| 项目 | 内容 |
|------|------|
| **模板** | 使用压缩工具将长 prompt 精炼为更短的等效版本。例如 LLMLingua-2 将 prompt 压缩 2-5 倍；Nano-Capsulator 压缩率达 ~81%；Prompt-SAW 利用关系图压缩至 35-57%。 |
| **原理** | 长 prompt 导致高延迟和高成本。LLMLingua-2 将压缩蒸馏为小型编码器（token 分类），运行速度比前代快 3-6 倍，端到端延迟降低 1.6-2.9 倍。Prompt-SAW 使用关系感知图保留关键信息。 |
| **来源** | LLMLingua-2（arXiv:2403.12968）225 引用、Nano-Capsulator（arXiv:2402.18700）、Prompt-SAW（arXiv:2404.00489），106 篇相关论文 |
| **适合任务** | 长文档处理、RAG 上下文、API 成本优化、低延迟推理 |

---

## 11. 提示重写（Prompt Rewriting）

| 项目 | 内容 |
|------|------|
| **模板** | 将初始 prompt 通过专门训练的重写器进行优化。例如 PRewrite 使用 RL 训练 LLM 重写弱 prompt；SCULPT 将长 prompt 表示为树结构，通过 Critic-Actor 循环优化。 |
| **原理** | 非专业用户编写的 prompt 往往不够精确。通过 RL 或迭代优化将弱 prompt 重写为强 prompt，SCULPT 即使在对抗性扰动下也能稳定提升效果。 |
| **来源** | PRewrite（arXiv:2401.08189）、SCULPT（arXiv:2410.20788），31 篇相关论文 |
| **适合任务** | 用户 prompt 质量不可控的场景、对话系统、长 prompt 优化 |

---

## 12. 提示集成（Prompt Ensemble）

| 项目 | 内容 |
|------|------|
| **模板** | 对同一任务使用多个不同 prompt 变体，汇总结果取最优或投票。例如 MCS-SQL 使用多个 prompt 生成 SQL 候选，再通过置信度过滤和多选投票选出最佳。PromptEval 估计约 100 个模板变体的性能分布。 |
| **原理** | 单一 prompt 的表现不稳定。通过多 prompt 集成利用多样性降低方差。MCS-SQL 在 BIRD 上达到 65.5% 执行准确率。PromptEval 仅用约 2 次单 prompt 评估的预算即可估计 ~100 个变体的性能分位。 |
| **来源** | MCS-SQL（arXiv:2405.07467）、PromptEval（arXiv:2405.17202），62 篇相关论文 |
| **适合任务** | Text-to-SQL、高精度输出场景、模型评估、最佳 prompt 搜索 |

---

## 13. 角色/人设提示（Role & Persona Prompting）

| 项目 | 内容 |
|------|------|
| **模板** | 设定角色激活领域知识。例如：`你是一位资深{领域}专家，拥有20年经验。请以你的专业视角分析以下问题：{任务描述}` |
| **原理** | 角色设定触发模型的领域知识检索能力。但研究发现翻译领域中人际交互概念（如翻译简介、译者人设）只能适度帮助 ChatGPT，需要为人机交互开发新的 prompt 理论。跨语言研究表明礼貌程度对 LLM 性能有影响，但最佳水平因语言和文化而异。 |
| **来源** | arXiv:2403.00127、arXiv:2402.14531（54 引用），163 篇相关论文 |
| **适合任务** | 专业领域问答、翻译、内容创作、特定风格生成 |

---

## 14. 情感刺激提示（Emotional Stimulus Prompting）

| 项目 | 内容 |
|------|------|
| **模板** | 在 prompt 中加入情感激励或社会压力。例如：`这个任务对我的职业生涯非常重要，请务必认真对待。{任务描述}` 或结合自动生成的情感刺激 + 结构化框架。 |
| **原理** | 情感刺激可激活模型的"注意力"机制。APGP 框架自动融合情感刺激和结构化脚手架，在复杂问题求解上优于单独使用刺激或框架。System 1 和 System 2 认知过程相关的提示可减少社会偏见。 |
| **来源** | APGP（arXiv:2404.10500）、arXiv:2404.17218，45 篇相关论文 |
| **适合任务** | 复杂问题求解、减少偏见、需要"认真对待"的高要求任务 |

---

## 15. 提示偏差校准（Prompt Bias Calibration）

| 项目 | 内容 |
|------|------|
| **模板** | 在推理时估计"仅 prompt"产生的偏置表征，从内部激活中减去。或使用 PPCL 方法在微调时对 prompt 扰动（同音替换、同义替换、改写）施加一致性正则化。 |
| **原理** | Prompt 本身会引入非平凡的偏差（梯度优化的 prompt 最严重）。去偏可减少基准虚增并将事实检索提升最多约 10 个百分点。PPCL 在字符级和语义级扰动下恢复大部分性能损失。 |
| **来源** | arXiv:2403.09963（25 引用）、PPCL（arXiv:2402.15833），93 篇相关论文 |
| **适合任务** | 事实知识提取、知识评估、公平性要求高的场景、鲁棒性测试 |

---

## 16. 提示敏感性评估（Prompt Sensitivity Assessment）

| 项目 | 内容 |
|------|------|
| **模板** | 使用 POSIX 等指标量化 prompt 变化对输出的影响：将原 prompt 替换为保持语义的变体，度量对数似然相对变化。 |
| **原理** | 更大模型或指令微调并不能可靠降低敏感性。但即使一个 few-shot 示例通常就能降低敏感性。模板修改对选择题影响最大，改写对开放生成影响最大。450 个实验表明无单一 prompt 占主导地位。 |
| **来源** | POSIX（arXiv:2410.02185）、arXiv:2412.14454、arXiv:2406.12334，134 篇相关论文 |
| **适合任务** | 模型评估、prompt 选择策略、生产 prompt 质量保证 |

---

## 17. 提示注入防御（Prompt Injection Defense）

| 项目 | 内容 |
|------|------|
| **模板** | **Spotlighting**：对不可信数据施加来源标记变换，使模型区分指令和数据。**StruQ**：将指令和数据分离为结构化通道，微调模型只遵循指令通道。**Signed-Prompt**：用签名标记可信指令段。 |
| **原理** | 间接 prompt injection 在工具集成 agent 中尤为危险（InjecAgent 基准显示 GPT-4 约 24% 被攻破）。Spotlighting 将注入成功率从 >50% 降至 <2%。StruQ 和 Signed-Prompt 通过结构化分离显著提升防御。 |
| **来源** | Spotlighting（arXiv:2403.14720）133 引用、StruQ（arXiv:2402.06363）207 引用、InjecAgent（arXiv:2403.02691）261 引用、Signed-Prompt（arXiv:2401.07612），77 篇相关论文 |
| **适合任务** | LLM 应用安全、RAG 系统、agent 系统、处理不可信外部输入的场景 |

---

## 18. 越狱攻防（Jailbreak Attack & Defense）

| 项目 | 内容 |
|------|------|
| **模板** | **防御**：RPO 优化轻量级防御后缀，加在 prompt 末尾。**攻击**：Rainbow Teaming 使用 quality-diversity 搜索生成多样化对抗 prompt；SequentialBreak 将恶意指令嵌入良性多步对话链中。 |
| **原理** | RPO 在 GPT-4 上将攻击成功率降至约 6%，Llama-2 上降至 0%。Rainbow Teaming 生成数百个 >90% 攻击成功率的 prompt。SequentialBreak 揭示了序列上下文防御的薄弱点。 |
| **来源** | RPO（arXiv:2401.17263）150 引用、Rainbow Teaming（arXiv:2402.16822）164 引用、SequentialBreak（arXiv:2411.06426），114 篇安全相关论文 |
| **适合任务** | 模型安全评估、红队测试、安全对齐训练数据生成 |

---

## 19. 提示窃取与保护（Prompt Stealing & Protection）

| 项目 | 内容 |
|------|------|
| **模板** | **攻击**：output2prompt 从正常输出反推 system prompt（无需 logits 或越狱）。PLeak 专攻 LLM 应用 prompt 泄露。**防御**：PromptKeeper 将泄露检测为假设检验，发现泄露时用 dummy prompt 重新生成。Prompt Obfuscation 用等效混淆版替换原始 prompt。 |
| **原理** | 即使是对齐模型也极易被简单手法提取 prompt。output2prompt 可零样本跨 LLM 迁移。PromptKeeper 提供鲁棒保护同时保持对话自然。混淆方案在保持功能等价的同时抵抗去混淆攻击。 |
| **来源** | output2prompt（arXiv:2405.15012）、PLeak（arXiv:2406.05823）127 引用、PromptKeeper（arXiv:2412.13426）、Obfuscation（arXiv:2409.11026），96 篇相关论文 |
| **适合任务** | 商业 prompt 保护、SaaS LLM 应用安全、知识产权防护 |

---

## 20. 软提示微调（Soft Prompt Tuning）

| 项目 | 内容 |
|------|------|
| **模板** | 在模型输入前添加可学习的连续向量（soft tokens），仅训练这些向量而非整个模型。LoPA 使用低秩分解构建实例级 soft prompt。SuperPos-Prompt 将每个 soft token 重参数化为多个冻结词汇嵌入的叠加。 |
| **原理** | Soft prompt tuning 参数量极小但效果可媲美全量微调。LoPA 在 NLU 和代码任务上匹配强 PEFT 和全微调。SuperPos-Prompt 在 GLUE/SuperGLUE 上大幅超越残差 soft prompt。GenPI 可将长 prompt 内化到模型中，推理时免去显式 prompt。 |
| **来源** | LoPA（arXiv:2405.15282）、SuperPos-Prompt（arXiv:2406.05279）、GenPI（arXiv:2411.15927），170 篇相关论文 |
| **适合任务** | 资源受限的模型适配、多租户服务、需要保持基座模型冻结的场景 |

---

## 21. 提示缓存与调度（Prompt Caching & Scheduling）

| 项目 | 内容 |
|------|------|
| **模板** | 通过嵌入相似度判断两个 prompt 能否共享缓存答案，蒸馏微调嵌入使 AUC 从 0.51 提升到 0.81。Preble 实现分布式 prompt 调度优化 LLM serving 吞吐量。 |
| **原理** | 现实场景中大量 prompt 语义相近，缓存可大幅降低重复推理成本。但通用语义嵌入不适合判断"可缓存性"，需专门训练。分布式调度可在多节点间优化 KV-cache 复用。 |
| **来源** | arXiv:2402.01173、Preble（arXiv:2407.00023）50 引用，88 篇相关论文 |
| **适合任务** | 高并发 LLM 服务、API 成本优化、重复性查询场景 |

## 22. 提示用于遗忘（Prompt for Unlearning）

| 项目 | 内容 |
|------|------|
| **模板** | ECO 通过 prompt 分类器加零阶优化的嵌入扰动，在推理时抑制特定知识。SPUL 学习可附加的 soft prompt，通过遗忘损失和效用损失平衡。 |
| **原理** | 无需重训模型即可实现"选择性遗忘"。ECO 支持 0.5B 到 236B 参数模型，规模增大不增加额外成本。SPUL 改善遗忘与保持效用的权衡。 |
| **来源** | ECO（arXiv:2406.07933）101 引用、SPUL（arXiv:2406.12038），40 篇相关论文 |
| **适合任务** | 隐私合规（GDPR 遗忘权）、去除有害知识、版权内容删除 |

## 23. 多语言/跨语言提示（Multilingual & Cross-Lingual Prompting）

| 项目 | 内容 |
|------|------|
| **模板** | **CoTR**：在一个 prompt 中先将低资源语言翻译为高资源语言，执行任务，再翻译回。例如：`Step 1: Translate the following Marathi text to English. Step 2: Perform {task}. Step 3: Translate result back.` |
| **原理** | 低资源语言直接 prompting 效果差。CoTR 在马拉地语任务上显著提升，特别是仇恨言论检测。跨语言礼貌度研究表明不礼貌通常损害性能，但最优礼貌水平因语言文化而异。 |
| **来源** | CoTR（arXiv:2409.04512）、arXiv:2402.14531（54 引用），81 篇相关论文 |
| **适合任务** | 低资源语言 NLP、跨语言翻译、多语言分类 |

---

## 24. 代码生成流程化提示（Code Generation Flow Engineering）

| 项目 | 内容 |
|------|------|
| **模板** | 将代码生成从"单 prompt"转为测试驱动的多阶段迭代流程：规格 → 测试生成 → 代码生成 → 测试验证 → 修复。AlphaCodium 将 GPT-4 在 CodeContests 上的 pass@5 从 19% 提升到 44%。 |
| **原理** | 单次 prompt 无法处理复杂编程任务。通过将问题分解为规格制定、测试设计、迭代修复等步骤，每步聚焦一个子目标，显著提升代码正确率。 |

## 29. 自一致性提示（Self-Consistency Prompting）

| 项目 | 内容 |

| **原理** | 单次采样可能落入错误推理路径。多次采样利用多样性和多数决降低方差。OpenMedLM 纯靠 prompt engineering 就在 MedQA 等关键医学基准上超越了微调过的开源 SOTA。 |

## 31. 免提示推理（Reasoning Without Prompting）

## 36. 提示分解攻击（Prompt Decomposition Attack）

| 项目 | 内容 |
|------|------|
| **模板** | **DrAttack**：将有害请求拆分为多个看似无害的子 prompt，通过 ICL 和同义词替换重组。仅需约 15 次查询就在 GPT-4 上达到约 78% 攻击成功率。 |
| **原理** | 安全对齐主要针对完整有害指令。将指令分解后每个片段看起来无害，绕过安全检查后再由模型隐式重组。这种方式比传统 prompt 攻击查询量少得多。 |

## 38. 提示敏感性诊断（Prompt Sensitivity Diagnostics）

| 项目 | 内容 |
|------|------|
| **模板** | **ProSA**：PromptSensiScore + 解码置信度分析，实例级 prompt 变异评估。配合两个无标签指标 — **sensitivity**（改写后预测翻转率）和 **consistency**（同类改写稳定性）。 |
| **原理** | 敏感性因数据和模型而异，更大模型和更高置信度趋于更鲁棒，few-shot 有帮助，主观评估在困难推理任务上特别敏感。这些指标无需标签即可指导 prompt 设计。 |

## 39. Text-to-SQL 增强提示（Prompt-Enhanced Text-to-SQL）

| 项目 | 内容 |
|------|------|
| **模板** | **PET-SQL**：第一轮用参考增强 schema prompt 生成 PreSQL 进行 schema 链接缩小上下文，第二轮生成最终 SQL，跨不同 LLM 做 cross-consistency 精炼。 |

## 组合应用方案

### 方案 A：高精度推理组合

```
CoT (技巧1) + 分隔符增强 (技巧2) + 提示集成 (技巧12)
```

先用 CoT 引导推理，分隔符提升多示例效果，多 prompt 投票确保稳定性。

### 方案 B：生产级安全部署组合

```
提示注入防御 (技巧17) + 提示窃取保护 (技巧19) + 提示缓存 (技巧21)
```

Spotlighting 防注入 + PromptKeeper 防泄露 + 缓存降本。

### 方案 C：Prompt 自动化流水线

```
自动优化 (技巧9) + 提示压缩 (技巧10) + 敏感性评估 (技巧16)
```

自动搜索最优 prompt → 压缩降低成本 → 评估鲁棒性。

### 方案 D：复杂任务分治组合

```
元提示 (技巧7) + 提示链 (技巧6) + 代码流程化 (技巧24)
```

Meta-Prompting 分解任务 → Chaining 逐步执行 → 测试驱动迭代。

### 方案 E：低资源 / 冷启动场景

```
自动演示生成 (技巧4) + 跨语言提示 (技巧23) + 混合专家 (技巧5)
```

利用模型自身生成示例 + 翻译桥接低资源语言 + MoE 覆盖多样化输入。

### 方案 F：全栈安全防御组合

```
偏好对齐防御 (技巧37) + Spotlighting (技巧17) + 提示混淆 (技巧19) + 多Agent安全 (技巧35)
```

SecAlign 对齐层防御 + Spotlighting 数据标记 + prompt 混淆防泄露 + AgentDojo 验证。

### 方案 G：医学/高精度问答组合

```
自一致性 (技巧29) + CoT (技巧1) + 角色设定 (技巧13) + 偏差校准 (技巧15)
```

CoT 推理 + kNN 示例选择 + 医学专家角色 + 自一致性投票 + 去偏。OpenMedLM 证明此组合超越微调 SOTA。

### 方案 H：数据库智能查询组合

```
Text-to-SQL (技巧39) + 提示集成 (技巧12) + 提示链 (技巧6)
```

两轮渐进 prompt（PET-SQL）+ 多 prompt 候选 + 跨模型一致性投票。

### 方案 I：训练数据工厂组合

```
合成数据 (技巧26) + 因果去偏 (技巧27) + 敏感性诊断 (技巧38)
```

Magpie 大规模生成 → 因果去偏过滤 → ProSA 验证质量。
